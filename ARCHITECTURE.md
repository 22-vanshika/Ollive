## Ingestion Flow

Every LLM call in Ollive goes through a five-step pipeline before a log entry reaches the database.

```
Browser SDK
    │
    ├─ ① Intercept — loggedLLMCall() wraps the provider adapter call
    │       Records timestamp_request, starts a latency timer
    │
    ├─ ② Execute — provider-adapter.ts streams the NDJSON response
    │       Accumulates token counts from the stream's usage chunks
    │       Calls onChunk(token) to update the UI incrementally
    │
    ├─ ③ Measure — finally{} block fires on both success and error paths
    │       latency_ms = Date.now() - requestStart
    │       Captures the last user message as input_preview
    │       Captures the full assistant response as output_preview
    │
    ├─ ④ Redact — redactPII() runs on both previews
    │       Strips: email, phone, SSN, card numbers, API keys, Bearer tokens
    │       Previews are also capped at 200 characters
    │
    └─ ⑤ Ship — navigator.sendBeacon(url, blob)
            Fire-and-forget, survives page unload
            Falls back to fetch() if the beacon queue is full
            Any failure is silently swallowed — log shipping never surfaces errors to users
```

**On the backend (POST /api/v1/ingest):**

```
HTTP request arrives
    │
    ├─ Pydantic validation
    │       Rejects malformed payloads with 422
    │       AwareDatetime rejects naive (timezone-less) timestamps
    │       status must be exactly "success" | "error" | "timeout"
    │
    ├─ Idempotency check
    │       SELECT by request_id (unique index, fast)
    │       If found → 409 Conflict, no write
    │       Same request_id from a retry is safely ignored
    │
    ├─ Defensive PII redaction (second pass)
    │       Runs the same regex rules as the SDK
    │       Guarantees no raw PII reaches the DB regardless of SDK version
    │
    └─ Async write
            asyncpg INSERT into inference_logs
            Returns 201 with { id, request_id, status }
```

## Logging Strategy

**What is logged — and what isn't.**

Ollive logs *metadata about inference calls*, not full conversation payloads. Storing complete prompts and responses at scale is expensive (storage, cost, compliance risk). Instead:

| Field | What's stored | Why |
|---|---|---|
| `input_preview` | First 200 chars of user message, PII-redacted | Enough to understand what triggered the call |
| `output_preview` | First 200 chars of assistant response, PII-redacted | Enough to audit quality |
| `latency_ms` | End-to-end time including network, not just TTFT | Captures the full user-perceived delay |
| `prompt_tokens` / `completion_tokens` | From the provider's usage object in the stream | Accurate cost attribution |
| `status` | `success` / `error` / `timeout` | Error rate calculation |
| `error_code` | Provider error message on failure | Root-cause diagnostics |
| `model` | Exact model string from the provider | Detects if Groq serves a different variant |

**Full messages** are stored separately in the `messages` table (for conversation resume), but that table is user-facing data, not observability data — it has no metrics aggregation.

**Latency classification:**
The SDK classifies a failure as `timeout` if `err.message.includes("timeout")` (triggered by `AbortSignal.timeout(30_000)`), and `error` for all other failures. This distinction matters for the dashboard: network timeouts and API rate limits have different operational responses.

**Idempotency:**
`request_id` is a UUID generated by the SDK before the LLM call is made. If `sendBeacon` fails and the browser retries, the second POST carries the same `request_id`. The backend's uniqueness check rejects it with 409 — the log is written exactly once.

## Scaling Considerations

### Write path (ingestion)

**Current:** Direct synchronous HTTP POST → asyncpg write. Works well up to ~500 RPS on a single Railway instance. SQLAlchemy's async session pool handles concurrent connections without thread blocking.

**Bottleneck:** At high sustained write volume, the database connection pool becomes the constraint (Supabase PgBouncer helps significantly). A write-heavy burst can cause ingestion latency to spike.

**Upgrade path:**
```
SDK → sendBeacon → Redis Streams (producer)
                        │
                   Consumer worker
                        │
                   Batch INSERT (100 rows / 500ms)
                        │
                   inference_logs
```
Batch inserts are 10–50× more efficient than single-row inserts at scale. The consumer worker decouples ingestion latency from database capacity.

### Read path (dashboard)

The three indexes on `inference_logs` cover all current dashboard queries:
- Summary stats (`AVG`, `SUM`, `COUNT`) over the full table — no index helps here; acceptable at <100k rows, needs TimescaleDB or ClickHouse beyond that
- Latency time series — filtered by `created_at` range; a `(created_at DESC)` index would help at >1M rows
- Recent logs feed — `ORDER BY created_at DESC LIMIT 50` — fast with the existing setup

**Upgrade path for analytics at scale:** Move `inference_logs` to TimescaleDB (PostgreSQL-compatible, automatic time partitioning) or ClickHouse (columnar, sub-second GROUP BY over billions of rows). The application layer is unchanged — only the connection string and query driver change.

### Horizontal scaling

The FastAPI backend is stateless (all state in PostgreSQL, no in-process session cache). Horizontal scaling is a matter of adding replicas behind a load balancer. Railway supports this with a single config change.

The frontend is static files served by Vercel's CDN — it scales automatically.

## Failure Handling

### LLM provider failures

```python
except groq.APITimeoutError:   → ProviderError → HTTP 502
except groq.AuthenticationError: → ProviderError → HTTP 502
except groq.RateLimitError:    → ProviderError → HTTP 502
except groq.APIError:          → ProviderError → HTTP 502
```

All provider failures are caught and mapped to a structured `ProviderError` exception. The global exception handler in `main.py` converts it to a JSON `{"detail": "..."}` response. The SDK logs the failure as `status: "error"` with the error message as `error_code`.

### Database failures

Repository functions use `async with session` via FastAPI's `Depends(get_db)`. If a write fails, SQLAlchemy rolls back the transaction automatically. The `InferenceLog` row is not partially written. The client receives a 500 with no internal detail exposed.

### Network failures (SDK → ingestion)

`sendBeacon` is best-effort. If the beacon fails (browser offline, OS queue full):
- The `fetch` fallback fires
- If the fetch also fails, the error is silently caught (`catch(() => {})`)
- The log entry is lost for that call

**Acceptable tradeoff for observability data.** A missed log is not a missed transaction. In v2, an IndexedDB retry queue with exponential backoff would recover logs from short network interruptions.

### Duplicate delivery

The SDK can deliver the same log twice (page refresh mid-send, service worker retry). The `request_id` unique constraint catches it: the second insert returns a 409 which the SDK ignores. The database always contains exactly one row per inference call.

### Conversation deletion during an active session

If a user deletes a conversation while messages are being streamed:
- Frontend: the active conversation ID is cleared; new messages have no target and are discarded
- Backend: the `messages` cascade delete removes all rows; `inference_logs.conversation_id` is set to `NULL` (not deleted) — the telemetry is preserved

## What Was Not Built (and Why)

### Event-based architecture

The current HTTP ingestion path is synchronous: the SDK fires a request, the backend writes, done. An event-based approach (SDK → Kafka topic → consumer → DB) would decouple write throughput from database capacity, enable fan-out to multiple consumers (alerting, archival, real-time aggregation), and add replay capability.

**Why it was skipped:** This is infrastructure, not application code. A Kafka or Redis Streams cluster adds operational complexity that is hard to justify before there's a measured bottleneck. The current architecture is designed so that adding a queue layer is a contained change — the ingestion service becomes a consumer, and the existing `ingest_log()` business logic is unchanged.

### Self-hosted Kubernetes

The application runs on Railway (backend) and Vercel (frontend) — both are PaaS platforms that handle infrastructure. Kubernetes manifests would enable self-hosted deployment with fine-grained resource control and horizontal pod autoscaling, but they also require maintaining a cluster, writing manifests for every service, and operating a container registry pipeline.

**What it would take:**
- `Deployment` + `Service` for backend and frontend
- `Ingress` with TLS termination
- `HorizontalPodAutoscaler` on the backend
- `ConfigMap` + `Secret` for environment configuration
- A `CronJob` or init container for Alembic migrations
- A PostgreSQL `StatefulSet` (or external managed DB)
